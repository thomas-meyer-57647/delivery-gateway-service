1. Zweck und Zielbild
Der Delivery-Gateway-Service ist die technische Ausführungsschicht für externe Zustellung:
nimmt konkrete Zustellversuche (Delivery Attempts) per REST entgegen,
persistiert Attempt + Timeline (Events),
führt die Zustellung über Provider/Adapter aus (z. B. SMTP, SMS/WhatsApp Provider),
nimmt Provider-Callbacks entgegen, mappt sie auf interne Event-Typen und aktualisiert den Attempt-Status,
stellt Status + Timeline für Support/Folgeprozesse bereit.
Wichtig (Abgrenzung): Der Service ist kein Orchestrator. Er entscheidet nicht über Policies, Reihenfolge, Fallbacks oder “welcher Kanal als nächstes”. Das liegt beim aufrufenden Orchestrator-Service (z. B. messaging-service oder communication-service, je nach Plattform-Setup).
2. Begriffe (Glossar)
Attempt (DeliveryAttempt): ein konkreter Zustellversuch für einen Kanal (EMAIL/SMS/WHATSAPP/VOICE) inkl. Empfänger(n) und Payload.
Event (DeliveryEvent): zeitlich geordnetes Zustellereignis (QUEUED, SENT, DELIVERED, FAILED …) als Timeline.
ProviderMessageId: ID des Providers für spätere Callback-Zuordnung.
Idempotency-Key: Header zur Wiederholungssicherheit (kein Doppelversand bei Retries).
Tenant / companyId: Mandant. Alle Datenzugriffe sind strikt an companyId gebunden.
CorrelationId: Request-Korrelation (Header X-Correlation-Id) für verteiltes Logging/Tracing.

2. Scope / Nicht-Scope
2.1 Im Scope
REST API für Attempt-Annahme, Statusabfrage, Provider-Callbacks
Persistenz in MariaDB/MySQL (Attempts, Events, optional Outbox)
Provider-Adapter (SMTP, SMS/WhatsApp Provider)
Mapping Provider-Event → internes Statusmodell
Multi-Tenant-Isolation (companyId aus JWT)
2.2 Nicht im Scope (wird absichtlich nicht in D implementiert)
Orchestrierung/Policy/Fallback/strategische Retries (liegt beim Orchestrator-Service)
Inbox/In-App Zustellung
Webhook-Delivery-Engine für Fremdsysteme (liegt bei einem dedizierten Webhook-Service)
Template-Rendering / Content-Composition
Eigenes File-Storage

3. Inter-Service-Schnittstellen (klar & mit exakten Bezeichnungen)
3.1 Was andere Services von D erhalten (D als Anbieter)
(A) Orchestrator-Service → Delivery-Gateway (Attempts & Status)
POST /api/v1/deliveries
Legt einen Attempt an (und startet ggf. die Zustellung).
Response: 202 mit DeliveryReceipt (inkl. attemptId)
GET /api/v1/deliveries/{attemptId}
Liefert Attempt + Timeline (Events) für Support/Weiterverarbeitung.
Response: 200 mit AttemptStatusResponse
Optional (nur wenn du diese Steuerung wirklich in D haben willst; Standard ist “Retry via neuem Attempt durch Orchestrator”):
POST /api/v1/deliveries/{attemptId}/retry
POST /api/v1/deliveries/{attemptId}/cancel
(B) Provider → Delivery-Gateway (Callbacks)
POST /api/v1/providers/{provider}/events
Nimmt Provider-Statusereignisse entgegen und persistiert sie als DeliveryEvent.
Response: 202 mit DeliveryEventReceipt (oder minimalem Ack)
(C) Betriebsendpunkte (Ops/Monitoring)
GET /actuator/health
GET /actuator/info
GET /swagger-ui
GET /v3/api-docs


3.2 Was D von anderen Services/Systemen holt (D als Konsument)
(A) Authorization Server (JWT/JWKS)
D validiert JWTs über die konfigurierte JWKS-URL:
ENV/Property: JWT_JWK_SET_URI → spring.security.oauth2.resourceserver.jwt.jwk-set-uri
(B) Externe Provider
EMAIL: SMTP Server (Host/Port/Username/Password)
SMS/WHATSAPP: Provider HTTP APIs über Adapter (z. B. Twilio/Vonage/Meta)
(C) Attachments (zwei zulässige Modi; beides ist implementierbar)
1. Inline (klein): Attachment kommt als Base64 im Attempt-Request → D muss nichts extern laden.
2. Referenz (skalierbar): Attachment kommt als fileId oder url → D lädt beim Versand:
GET /api/v1/files/{fileId}/download (falls du einen File-Service so benennst)
oder direkte attachments[].url (pre-signed URL, z. B. S3/MinIO)
Wenn du “später sauber skalieren” willst: Referenzmodus ist der Standard, Inline bleibt optional für kleine Dateien.
(D) Interne Push-Weiterleitung (bewusst nicht fest verdrahtet in D)
D ruft keine internen Services “ungefragt” zurück, solange kein dedizierter Webhook-Delivery-Service als Ziel festgelegt ist.
Status/Events werden primär über Pull konsumiert (GET /api/v1/deliveries/{attemptId}).

4. Sicherheits- und Multi-Tenant-Festlegungen
4.1 Authentifizierung / Autorisierung
Alle fachlichen Endpunkte sind JWT-geschützt (außer Health/Info/Docs).
Tenant wird aus JWT-Claim companyId abgeleitet.
Jeder DB-Zugriff wird stets mit Filter companyId = <claim> ausgeführt.
Cross-Tenant Zugriff ergibt 404/403 (keine Informationslecks).
4.2 Provider Callback Security
Provider-Callbacks werden je Provider abgesichert (Signatur/HMAC Header oder vergleichbar).
Mapping “welcher Provider nutzt welche Signatur” ist in Adapter-Konfiguration dokumentiert.
5. Statusmodell
5.1 Attempt-Status
QUEUED – angenommen, wartet auf Verarbeitung
PROCESSING – Versand läuft
SENT – Provider hat angenommen/gesendet (nicht zwingend zugestellt)
DELIVERED – zugestellt bestätigt (wenn Provider das liefern kann)
FAILED – final fehlgeschlagen
CANCELED – abgebrochen
5.2 Event-Typen (Timeline)
Mindestens:
QUEUED, PROCESSING, SENT, DELIVERED, FAILED, CANCELED
Optional, wenn sauber definierbar:
BOUNCED, REJECTED, UNDELIVERABLE, READ

6.5 GET /api/v1/wallet – WalletBalanceResponse
Header
Authorization: Bearer <JWT>
X-Correlation-Id (optional, wird durchs Request-Handling durchgereicht)
Response 200
{
  "companyId": "company-a",
  "balance": 42
}
Hinweis: Der Header `companyId`-Claim steuert die Mandantenzuordnung; fehlt er, antwortet D mit 403, ohne Zugriff auf andere Mandanten.

6.6 POST /api/v1/wallet/topups – WalletTopupRequest
Header
Authorization: Bearer <JWT>
X-Correlation-Id (optional)
Body
{
  "amount": 5
}
Response 200 – WalletBalanceResponse
{
  "companyId": "company-a",
  "balance": 47
}
Fehlerfälle
- 400 bei fehlendem oder kleinem `amount` (<=0)
- 401 ohne JWT
- 403 ohne `companyId`-Claim
Topups schreiben einen Ledger-Eintrag vom Typ `TOPUP` in `wallet_ledger` und aktualisieren `tenant_wallet.balance`.

6.4 Idempotenz (praktisch, implementierbar)
Es gibt zwei Ebenen (beides kompatibel):
1. Header Idempotency-Key (empfohlen)
Unique: (companyId, idempotencyKey)
Wiederholung liefert denselben Receipt, ohne zweiten Versand.
2. attemptId (wenn vom Orchestrator vorgegeben)
Unique: (companyId, attemptId)
Wiederholung liefert denselben Receipt.
Implementationshinweis: du setzt mindestens eine Unique-Regel in der DB. Viele Teams setzen beide, wobei Idempotency-Key als “Retry-Schutz” der bessere Standard ist.

8. Verarbeitung / Ablauf (intern)
8.1 Synchronous oder Asynchronous
D kann Versand synchron (direkt im Request) oder asynchron (Worker) ausführen.
Bei asynchron:
Attempt wird initial QUEUED,
Worker setzt PROCESSING und schreibt Events,
final SENT/DELIVERED/FAILED.
8.2 Fehlerklassifikation
transient/retryable (Timeout, Netzwerk, Provider 5xx)
non-retryable (ungültiger Empfänger, 4xx, Policy Violation)
Strategische Retries/Fallbacks bleiben beim Orchestrator; D kann höchstens kurze technische Retries machen (1–2), und protokolliert jeden Versuch als Event.

9.3 Credits / Wallet
Service D führt ein einfaches Prepaid-Wallet pro Tenant:
- `GET /api/v1/wallet` gibt `{companyId,balance}` zurück und benötigt einen validen JWT mit `companyId`.
- `POST /api/v1/wallet/topups` akzeptiert `{amount}` (positiv) und bucht den Betrag; historisierte Ledger-Einträge (`wallet_ledger`) protokollieren Topups und spätere Debits.
> Das Wallet speichert `tenant_wallet.balance` + `wallet_ledger` (TOPUP/DEBIT) und alle Buchungen bleiben im selben Tenant.
- Bei `POST /api/v1/deliveries` fallen sms-/whatsapp-spezifische Kosten an (`recipientsCount * COST`). Ist der Saldo zu gering, antwortet D mit `409 INSUFFICIENT_CREDITS` und schreibt keine Attempt/Events.
- Das Wallet wird pro Tenant über ENV-Variablen `CREDITS_ENABLED`, `SMS_COST`, `WHATSAPP_COST`, `EMAIL_COST` gesteuert (Defaultwerte siehe oben). Wird `CREDITS_ENABLED=false` gesetzt, entstehen keine Debits; Rechnungsdaten bleiben erhalten (Ledger).
- Idempotente Versuche (per `attemptId` oder `Idempotency-Key`) dürfen die Salden nur einmal belasten, weil `(companyId, attemptId)` und `(companyId, idempotency_key)` unique sind.

10. Observability / Fehlerantworten
X-Correlation-Id wird übernommen oder generiert und in Response zurückgegeben.
Strukturierte Logs mit companyId, attemptId, correlationId.
Standard-ErrorResponse (JSON) enthält mindestens: timestampUtc, status, error, message, path, correlationId.

12. Implementierungs-Checkliste
